# Today's lecture

## 📚 Deep Dive: Class vs. Instance (OOP)

**Scenario:**
> You want to bring a "Real World" entity into the "Program Domain." Physical objects cannot exist inside a computer's memory directly.



**Q: What is the mechanism used to bridge the real world and the digital program?**

*   **Class (The "Template"):** A blueprint or template defined by `{}`. It analyzes real-world objects into **Variables** (noun-like features/properties) and **Methods** (verb-like actions/behaviors).
*   **Instance (The "Operating Object"):** The actual object created from the template that lives and works within the program's memory. While a "Class" represents the real-world concept, an "Instance" is the specific manifestation running in the software.

## 🛠️ Environment Configuration

*   **Symptom/Task:** Transitioning from a code editor to a functional execution environment.
*   **Context/Environment:** Using VS Code as an IDE vs. installing the JDK.
*   **Root Cause/Logic:** VS Code is a code editor (writing tool) and lacks the execution environment required by Java.
*   **Resolution/Action:** Install the **JDK** to establish the **JVM** layer on top of the Operating System.
*   **System Rule (Takeaway):** Java software is **Independent** of the OS platform but **Dependent** on the JVM.

## 📚 Additional Knowledge: Platform Independence



**Scenario:**
> Running Java software on different hardware setups and Operating Systems.

**Q: Why is Java considered "Write Once, Run Anywhere" (WORA)?**

**[OS Independence]:** In traditional systems, software is OS-dependent. In Java, the **JDK** installation creates a **Virtual Machine (JVM)**. The application runs on this virtual layer rather than the OS directly, allowing it to function on any system that has a JVM installed.

## 📚 Deep Dive: Bytecode vs. Binary Code

**Scenario:**
> After compiling a Java file, you obtain a file consisting of 0s and 1s. While it looks like machine code, its dependency defines its classification.

**Q: What is the technical distinction between Bytecode and Binary Code in the Java ecosystem?**

**Bytecode (The "VM-Dependent Code"):** The 0s and 1s found in a `.class` file. It is a specialized format that is dependent on the Virtual Machine (JVM), allowing the program to remain platform-independent.

**Binary Code (The "OS-Dependent Code"):** Traditional machine code that is directly dependent on the specific Operating System (OS). Unlike Bytecode, it cannot run across different platforms without being recompiled for each OS.

## 🛠️ System Rule: Case Sensitivity

Java is a **case-sensitive** language. The compiler recognizes uppercase and lowercase letters as distinct characters.

*   **Logic:** `Teacher` and `teacher` are treated as entirely different identifiers.
*   **Takeaway:** Precision in capitalization is required to avoid errors in the program domain.

## 📚 Deep Dive: camelCase (The "Camel" Style)

**Scenario:**
> When naming identifiers, multiple words are often combined (e.g., "English" and "Sum").

**Q: Why is this naming style called "Camel Case" in Java?**

**Visual Logic (The "Hump"):** The name comes from the visual appearance of a camel's back. While the first word starts with a lowercase letter, every subsequent word starts with a capital letter, creating "humps" in the middle of the identifier.

**Functional Use (Readability):** This ensures that even though variables like `engSum` are noun-like characteristics, the separation between words is clear without using spaces or special characters.
